Purchase Order Automation

MVP SOW Business Rules & Technical Implementation Overview (Updated)

1. High-Level Business Overview (MVP)

The primary objective defined in the project is to stand up a minimal viable product (MVP) MuleSoft solution to ingest and parse regular and blanket purchase orders, including up to 20 variations, establishing an integration across MuleSoft and Salesforce, and defining PO reprocessing and resend rules for exception handling.

The system is designed to eliminate manual data entry by extracting critical data from incoming PDF documents and validating them against the existing CRM records.

  • Ingestion: Seamless fetching of PO PDFs deposited in AWS S3 by external vendors or systems.
  • Extraction: Utilizing AI-driven MuleSoft Intelligent Document Processing (IDP) to parse unstructured PDFs into structured data.
  • Validation: Real-time cross-referencing of extracted financial and logistical data with the associated Salesforce Opportunity (utilizing FSC Standard Objects).
  • Resolution: Touchless automatic approval of matching POs, or intelligent routing to a manual review queue when discrepancies are found.

3. AWS S3 Integration

⚠️ Pending Client Definition The exact mapping of S3 bucket names per environment (Dev/QA/UAT/Prod) and the mandatory file naming convention (e.g., po_file_*.pdf) are pending final client confirmation.

Connection Configuration:

  • Authentication: Relies on the default AWS credentials chain (tryDefaultAWSCredentialsProviderChain="true").
  • Cross-Account Access: Assumes a specific IAM Role via STS (<s3:role arn="${aws.s3.roleArn}" />) to grant access to Twist Bioscience's secure buckets.
  • Region: Configured dynamically via environment properties.

Key Operations:

  • s3:ListObjects Used by both the Scheduler and System API to locate files.
  • s3:GetObject Used by the Consumer flow to download the binary PDF.

4. Salesforce Integration

Architectural Decision Record To align with the actual Twist FSC schema, the validation target was updated from the non-existent PurchaseOrder/Quote objects to the standard Opportunity object.

Connection Configuration:

  • Authentication: OAuth 2.0 JWT Bearer Token flow.
  • Principal Account: mulesoftapi@twistbioscience.com
  • Security: Keystore (keystore.p12) signed payload.

SOQL Query (Consulted Fields):

SELECT Id, StageName, Amount
FROM Opportunity
WHERE Name = :PoNumber

5. Business Rules & Update Conditions (MVP)

In this MVP phase, due to standard object schema limitations, the matching logic strictly relies on verifying the exact monetary amount of the identified Opportunity.

MuleSoft Data (IDP Extraction) Salesforce CRM Field Mapping Validation Condition
poNumber Opportunity.Name Used as the primary key for the initial SOQL Query.
totalAmount Opportunity.Amount Must match exactly.

Action Outcomes based on MVP Validation:

IF SUCCESS (Data Matches)

  • Action: salesforce:update (Type: Opportunity)
  • Update Fields: StageName = "Closed Won"

IF ERROR (Data Mismatch OR System Error)

  • Action 1: salesforce:update (Type: Opportunity)
  • Update Fields: StageName = "Negotiation/Review"
  • Action 2: salesforce:create (Type: Task)
  • Task Data: Subject = "Manual Review Required...", Status = "Not Started", Priority = "High"
  • Fail-safe: MuleSoft Global On-Error-Continue ensures processing continues for the next queued file.

6. Future Improvements and Considerations

Atrium's Recommendations for building a scalable solution beyond the MVP phase include:

  • Communication Suppression: Timing of the automated job must be considered, specifically regarding how to prevent the system from sending call-to-action emails for orders currently being processed by the automation (i.e., how to suspend these communications while the job is running).
  • Touchless Exclusions: The need to ensure the automation does not conflict with existing "touchless" flow exclusions.
  • Validation Sandbox: Lack a clear method to validate the process without interfering with daily workflows.
  • Approval Pipeline: Changing the verification status to "approved" should be the final step in the process to prevent downstream side-effects during intermediate processing.
